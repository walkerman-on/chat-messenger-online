# ОТЧЕТ ПО РЕАЛИЗОВАННОМУ ФУНКЦИОНАЛУ БЭКЕНД-СИСТЕМЫ

## ВВЕДЕНИЕ

Данный отчет описывает реализованный функционал бэкенд-системы веб-приложения для обмена сообщениями, разработанного на базе фреймворка NestJS. Система представляет собой полнофункциональный backend для мессенджера с поддержкой реального времени, системой аутентификации, управлением пользователями, чатами, сообщениями, задачами и настройками приватности.

Современные мессенджеры требуют не только базового функционала отправки сообщений, но и целого комплекса дополнительных возможностей: безопасная аутентификация пользователей, управление контактами и друзьями, организация групповых чатов, отслеживание статусов доставки сообщений, возможность редактирования и пересылки сообщений, а также интеграция с системой управления задачами. Все эти функции должны работать в режиме реального времени, обеспечивая мгновенную синхронизацию данных между всеми подключенными клиентами.

Система построена на архитектуре REST API с использованием WebSocket для обеспечения коммуникации в реальном времени. В качестве базы данных используется PostgreSQL - надежная реляционная база данных, которая обеспечивает целостность данных и поддержку сложных связей между сущностями. Для кэширования и управления очередями используется Redis, что позволяет оптимизировать производительность системы при работе с большим количеством пользователей. Приложение полностью контейнеризировано с использованием Docker, что упрощает развертывание и обеспечивает консистентность работы в различных окружениях - от разработки до production.

Данная система может быть использована как в корпоративной среде для внутренней коммуникации сотрудников, так и в качестве основы для публичного мессенджера. Особое внимание уделено безопасности данных, что особенно важно при работе с конфиденциальной информацией в корпоративных чатах с уровнями секретности.

## 1. АРХИТЕКТУРА СИСТЕМЫ

### 1.1. Технологический стек

Выбор технологий для разработки бэкенд-системы был основан на требованиях к производительности, масштабируемости и удобству разработки. Бэкенд-система разработана с использованием следующих технологий:

- **NestJS 10.3.0** - прогрессивный Node.js фреймворк для построения масштабируемых серверных приложений. NestJS был выбран благодаря его модульной архитектуре, встроенной поддержке TypeScript, и мощной системе dependency injection, которая упрощает разработку и тестирование кода. Фреймворк предоставляет готовые решения для работы с базами данных, WebSocket, валидацией данных и многими другими задачами.

- **TypeScript 5.3.3** - типизированный язык программирования для обеспечения надежности кода. Использование TypeScript позволяет выявлять ошибки на этапе компиляции, улучшает читаемость кода и упрощает рефакторинг. Статическая типизация особенно важна при работе с большими проектами, где несколько разработчиков работают над одним кодом.

- **PostgreSQL** - реляционная база данных для хранения структурированных данных. PostgreSQL была выбрана как надежная и функциональная база данных с поддержкой сложных запросов, транзакций и целостности данных. Она отлично подходит для систем с множественными связями между сущностями, такими как пользователи, чаты, сообщения и задачи.

- **TypeORM 0.3.17** - ORM (Object-Relational Mapping) для работы с базой данных. TypeORM позволяет работать с базой данных используя объектно-ориентированный подход, что делает код более понятным и поддерживаемым. ORM автоматически генерирует SQL запросы, управляет миграциями и обеспечивает типобезопасность при работе с данными.

- **Socket.IO 4.6.1** - библиотека для реализации WebSocket соединений. Socket.IO обеспечивает двустороннюю коммуникацию между клиентом и сервером в реальном времени, что критично для мессенджера. Библиотека автоматически обрабатывает переподключения, поддерживает fallback на long-polling и предоставляет удобный API для работы с комнатами и событиями.

- **JWT (JSON Web Tokens)** - механизм аутентификации и авторизации. JWT токены позволяют реализовать stateless аутентификацию, где сервер не хранит информацию о сессиях пользователей. Это упрощает масштабирование системы и позволяет легко работать с распределенными системами. Токены содержат всю необходимую информацию о пользователе и подписываются секретным ключом для предотвращения подделки.

- **bcrypt** - библиотека для хеширования паролей. Безопасное хранение паролей критично для любой системы. Bcrypt использует адаптивный алгоритм хеширования, который автоматически увеличивает сложность вычислений с ростом вычислительной мощности, что защищает от атак методом перебора.

- **Swagger** - инструмент для автоматической генерации API документации. Swagger позволяет автоматически создавать интерактивную документацию API на основе декораторов в коде. Это значительно упрощает работу фронтенд-разработчиков и тестировщиков, которые могут сразу видеть все доступные endpoints и тестировать их прямо из браузера.

- **Docker** - платформа для контейнеризации приложения. Docker позволяет упаковать приложение со всеми его зависимостями в контейнер, что обеспечивает одинаковое поведение в различных окружениях. Это упрощает развертывание и устраняет проблемы с различиями в окружениях разработки и production.

### 1.2. Архитектурные принципы

Архитектура системы построена на принципах, которые обеспечивают масштабируемость, поддерживаемость и надежность кода. Каждый принцип решает конкретные задачи и упрощает разработку:

- **Модульность** - каждый функциональный блок выделен в отдельный модуль (auth, users, chats, messages и т.д.). Это означает, что код, связанный с аутентификацией, находится в модуле auth, код для работы с пользователями - в модуле users, и так далее. Такое разделение позволяет легко находить нужный код, изолировать изменения и тестировать отдельные части системы независимо друг от друга. Например, если нужно изменить логику работы с сообщениями, разработчик знает, что весь связанный код находится в модуле messages.

- **Разделение ответственности** - четкое разделение между контроллерами (обработка HTTP запросов), сервисами (бизнес-логика) и репозиториями (работа с данными). Контроллеры отвечают только за прием HTTP запросов и возврат ответов, они не содержат бизнес-логику. Вся логика работы с данными находится в сервисах, которые могут быть переиспользованы в разных местах приложения. Репозитории отвечают только за работу с базой данных. Такое разделение упрощает тестирование - можно легко заменить репозиторий на мок для тестирования сервиса без обращения к реальной базе данных.

- **Dependency Injection** - использование встроенного механизма DI NestJS для управления зависимостями. Вместо того, чтобы создавать объекты напрямую в коде, зависимости передаются через конструктор. Это делает код более гибким и тестируемым. Например, если сервис сообщений зависит от сервиса пользователей, он получает его через конструктор, а не создает сам. Это позволяет легко заменить реальный сервис на мок при тестировании.

- **Guards и Decorators** - использование декораторов и гардов для аутентификации и авторизации. Guards - это специальные классы, которые проверяют права доступа перед выполнением запроса. Декораторы позволяют легко помечать endpoints как публичные или защищенные, получать текущего пользователя и т.д. Это делает код более декларативным и понятным - сразу видно, какие endpoints требуют аутентификации, а какие нет.

- **DTO (Data Transfer Objects)** - использование DTO для валидации входящих данных. DTO - это классы, которые описывают структуру данных, которые приходят от клиента. Они автоматически валидируются при получении запроса, что гарантирует корректность данных до того, как они попадут в бизнес-логику. Это предотвращает множество ошибок и делает код более надежным.

## 2. МОДУЛЬ АУТЕНТИФИКАЦИИ (AUTH MODULE)

### 2.1. Описание функционала

Модуль аутентификации является фундаментом безопасности всей системы. Он обеспечивает регистрацию новых пользователей, вход в систему, обновление токенов доступа и получение информации о текущем пользователе. Без надежной системы аутентификации невозможно обеспечить безопасность данных пользователей и предотвратить несанкционированный доступ.

Система использует JWT (JSON Web Tokens) для аутентификации с разделением на access и refresh токены. Это современный подход к аутентификации, который не требует хранения сессий на сервере. Когда пользователь входит в систему, он получает два токена: короткоживущий access token для доступа к API и долгоживущий refresh token для обновления access token без повторного ввода пароля. Такой подход обеспечивает баланс между безопасностью (короткое время жизни access token) и удобством использования (не нужно постоянно вводить пароль).

Модуль также интегрирован с системой двухфакторной аутентификации, что добавляет дополнительный уровень защиты для пользователей, которые хотят максимально обезопасить свой аккаунт. При включенной 2FA пользователю необходимо не только ввести правильный пароль, но и предоставить код из приложения-аутентификатора.

### 2.2. Реализованные endpoints

**POST /api/auth/register** - Регистрация нового пользователя

Этот endpoint позволяет новым пользователям создать аккаунт в системе. Процесс регистрации включает несколько важных шагов безопасности:

- Принимает данные: email, password, username, firstName, lastName. Все поля обязательны для заполнения и проходят валидацию на стороне сервера. Email проверяется на корректность формата, пароль должен соответствовать требованиям безопасности (минимальная длина, сложность).

- Валидирует уникальность email и username. Система проверяет, не существует ли уже пользователь с таким email или username в базе данных. Если пользователь с такими данными уже существует, возвращается ошибка 409 (Conflict), что предотвращает создание дублирующихся аккаунтов.

- Хеширует пароль с использованием bcrypt (10 раундов). Пароль никогда не хранится в открытом виде. Вместо этого создается криптографический хеш, который невозможно обратить обратно в исходный пароль. Даже если злоумышленник получит доступ к базе данных, он не сможет узнать реальные пароли пользователей. Bcrypt с 10 раундами обеспечивает высокий уровень безопасности при приемлемой производительности.

- Создает пользователя в базе данных. После всех проверок создается новая запись в таблице users со всеми предоставленными данными. Одновременно создаются настройки приватности по умолчанию для нового пользователя.

- Возвращает access и refresh токены, а также данные пользователя без пароля. Сразу после регистрации пользователь получает токены для доступа к системе, что позволяет ему начать работу без дополнительного входа. Важно отметить, что пароль никогда не возвращается в ответе, даже в захешированном виде, для максимальной безопасности.

**POST /api/auth/login** - Вход в систему

Этот endpoint обеспечивает безопасный вход пользователя в систему. Процесс входа включает несколько этапов проверки:

- Принимает email и password. Пользователь предоставляет свои учетные данные для входа. Email используется как идентификатор пользователя, так как он уникален в системе.

- Проверяет существование пользователя. Система ищет пользователя с указанным email в базе данных. Если пользователь не найден, возвращается общая ошибка "Invalid credentials" без указания, что именно неверно (email или пароль). Это предотвращает перебор email'ов злоумышленниками, так как они не могут узнать, существует ли пользователь с данным email.

- Валидирует пароль через bcrypt.compare. Даже если пользователь найден, необходимо проверить правильность пароля. Система сравнивает предоставленный пароль с сохраненным хешем используя функцию bcrypt.compare, которая безопасно сравнивает пароль с хешем без возможности восстановления исходного пароля. Если пароль неверен, также возвращается общая ошибка "Invalid credentials".

- Поддерживает двухфакторную аутентификацию (если включена). Если у пользователя включена двухфакторная аутентификация, после успешной проверки пароля система требует дополнительный код подтверждения. Это добавляет второй уровень защиты - даже если злоумышленник узнает пароль, ему понадобится доступ к устройству пользователя (телефону с приложением-аутентификатором) для входа в систему.

- Возвращает токены и данные пользователя. После успешной проверки всех данных пользователь получает новую пару токенов (access и refresh) и информацию о своем профиле. Токены позволяют пользователю обращаться к защищенным endpoints без необходимости повторного ввода пароля.

**POST /api/auth/refresh** - Обновление access токена
- Требует валидный refresh токен
- Генерирует новую пару токенов
- Обновляет время жизни токенов

**GET /api/auth/me** - Получение текущего пользователя
- Возвращает полную информацию о текущем аутентифицированном пользователе
- Использует JWT для идентификации пользователя

### 2.3. Технические детали

Система использует два типа токенов, каждый из которых выполняет свою роль в обеспечении безопасности:

- **Access Token** - короткоживущий токен (по умолчанию 15 минут), используется для доступа к защищенным ресурсам. Этот токен содержит информацию о пользователе и отправляется с каждым запросом к API в заголовке Authorization. Короткое время жизни (15 минут) означает, что даже если токен будет перехвачен злоумышленником, он сможет использовать его только в течение ограниченного времени. Это минимизирует потенциальный ущерб от утечки токена. После истечения срока действия токена пользователь должен использовать refresh token для получения нового access token.

- **Refresh Token** - долгоживущий токен (по умолчанию 7 дней), используется для обновления access токена. Этот токен хранится на клиенте и используется только для получения новых access токенов. Он не используется напрямую для доступа к API, что ограничивает его применение в случае утечки. Refresh token имеет более длительный срок жизни, чтобы пользователю не приходилось постоянно вводить пароль. Однако если refresh token будет скомпрометирован, злоумышленник сможет получать новые access токены в течение 7 дней, поэтому важно хранить refresh token в безопасном месте (например, в httpOnly cookie).

Токены содержат payload с информацией о пользователе:
```typescript
{
  email: string,      // Email пользователя для идентификации
  sub: string,        // ID пользователя (subject) - основной идентификатор
  username: string    // Имя пользователя для отображения
}
```

Эта информация позволяет серверу идентифицировать пользователя без обращения к базе данных при каждом запросе, что повышает производительность системы. Однако важно понимать, что эта информация не является секретной и может быть прочитана из токена (токены кодируются в base64, но не шифруются), поэтому в токенах никогда не хранятся чувствительные данные, такие как пароли или права доступа.

Пароли хранятся в захешированном виде с использованием алгоритма bcrypt с 10 раундами хеширования. Bcrypt - это адаптивный алгоритм хеширования, который специально разработан для хеширования паролей. Он использует функцию Blowfish и автоматически увеличивает сложность вычислений с ростом вычислительной мощности. 10 раундов означает, что функция хеширования применяется 2^10 (1024) раз, что делает процесс хеширования достаточно медленным для злоумышленников (затрудняет перебор паролей), но достаточно быстрым для легитимного использования. Это обеспечивает высокий уровень безопасности при сохранении приемлемой производительности системы.

## 3. МОДУЛЬ УПРАВЛЕНИЯ ПОЛЬЗОВАТЕЛЯМИ (USERS MODULE)

### 3.1. Описание функционала

Модуль управления пользователями является центральным компонентом системы, который отвечает за всю информацию о пользователях приложения. Этот модуль предоставляет функционал для работы с профилями пользователей: просмотр, редактирование, загрузка аватаров, управление статусом пользователя.

В современном мессенджере профиль пользователя - это не просто имя и фамилия. Пользователи хотят видеть информацию о своих коллегах: их должность, отдел, текущий проект, биографию. Это помогает лучше понимать контекст общения и находить нужных людей. Например, если пользователь ищет специалиста по определенному проекту, он может посмотреть профили пользователей и найти нужного человека по полю "текущий проект".

Модуль также управляет статусами пользователей (онлайн, офлайн, отошел), что позволяет другим пользователям видеть, доступен ли человек для общения в данный момент. Эта информация обновляется автоматически при подключении и отключении пользователя через WebSocket, что обеспечивает актуальность данных в реальном времени.

Загрузка аватаров позволяет пользователям персонализировать свой профиль, делая общение более персональным и узнаваемым. Система валидирует загружаемые файлы, проверяя их тип и размер, чтобы предотвратить загрузку вредоносных файлов или файлов слишком большого размера.

### 3.2. Модель данных пользователя

Сущность User содержит следующие поля:
- **id** (UUID) - уникальный идентификатор
- **email** (string, unique) - электронная почта
- **password** (string, hashed) - захешированный пароль
- **username** (string, unique) - уникальное имя пользователя
- **firstName** (string) - имя
- **lastName** (string) - фамилия
- **avatar** (string, nullable) - путь к файлу аватара
- **status** (enum: ONLINE, OFFLINE, AWAY) - статус пользователя
- **lastSeen** (timestamp, nullable) - время последнего посещения
- **bio** (string, nullable) - биография пользователя
- **position** (string, nullable) - должность
- **department** (string, nullable) - отдел
- **currentProject** (string, nullable) - текущий проект
- **twoFactorSecret** (string, nullable) - секрет для 2FA
- **twoFactorEnabled** (boolean) - флаг включения 2FA
- **isActive** (boolean) - флаг активности аккаунта

### 3.3. Реализованные endpoints

**GET /api/users** - Получение списка всех пользователей
- Возвращает список всех активных пользователей системы
- Используется для поиска и добавления в друзья

**GET /api/users/me** - Получение профиля текущего пользователя
- Возвращает полную информацию о текущем пользователе
- Использует декоратор @CurrentUser() для получения данных из JWT

**GET /api/users/:id** - Получение пользователя по ID
- Возвращает публичную информацию о пользователе
- Используется для просмотра профилей других пользователей

**PATCH /api/users/me** - Обновление профиля
- Позволяет обновить: firstName, lastName, bio, position, department, currentProject, status
- При обновлении пароля автоматически хеширует новое значение
- Валидирует входящие данные через DTO

**POST /api/users/me/avatar** - Загрузка аватара

Этот endpoint позволяет пользователям загружать свои фотографии профиля. Процесс загрузки включает несколько этапов безопасности и оптимизации:

- Принимает файл изображения (jpg, jpeg, png, gif). Система проверяет расширение файла перед сохранением, чтобы убедиться, что это действительно изображение. Это предотвращает загрузку потенциально опасных файлов (например, исполняемых файлов или скриптов), которые могут быть замаскированы под изображения. Поддерживаются только популярные форматы изображений, которые гарантированно безопасны для отображения в браузере.

- Максимальный размер файла: 10MB. Ограничение размера файла необходимо по нескольким причинам: во-первых, большие файлы занимают много места на диске сервера, во-вторых, загрузка больших файлов может замедлить работу сервера и занять много времени для пользователя, в-третьих, для аватара обычно достаточно изображения небольшого размера. Если пользователь попытается загрузить файл больше 10MB, запрос будет отклонен с соответствующим сообщением об ошибке.

- Сохраняет файл в директорию `./uploads/avatars`. Все аватары хранятся в отдельной директории, что упрощает управление файлами и позволяет легко настроить статическую раздачу файлов через веб-сервер (например, nginx). Это также упрощает резервное копирование и миграцию данных.

- Генерирует уникальное имя файла. Вместо сохранения файла с оригинальным именем система генерирует случайное имя из 32 символов. Это решает несколько проблем: предотвращает конфликты имен (если два пользователя загрузят файл с одинаковым именем), скрывает оригинальное имя файла (что может быть важно для приватности), и предотвращает атаки, связанные с использованием специальных символов в имени файла.

- Обновляет путь к аватару в профиле пользователя. После успешного сохранения файла система обновляет запись пользователя в базе данных, сохраняя путь к новому аватару. Если у пользователя уже был аватар, старый файл остается на диске (в будущем можно добавить автоматическое удаление старых аватаров для экономии места). Путь к аватару хранится в формате `/uploads/avatars/имя_файла`, что позволяет легко формировать полный URL для отображения изображения в интерфейсе.

**DELETE /api/users/me** - Удаление аккаунта
- Удаляет текущего пользователя из системы
- Каскадно удаляет связанные данные (чаты, сообщения, друзья)

### 3.4. Управление статусом пользователя

Управление статусом пользователя - важная функция для мессенджера, которая позволяет пользователям видеть, доступны ли их собеседники для общения в данный момент. Система автоматически обновляет статус пользователя при подключении/отключении через WebSocket, что обеспечивает актуальность информации в реальном времени.

- При подключении: статус устанавливается в ONLINE. Когда пользователь открывает приложение и устанавливает WebSocket соединение, система автоматически обновляет его статус в базе данных на ONLINE. Одновременно рассылается уведомление всем другим пользователям, которые могут быть заинтересованы в этом статусе (например, друзья или участники общих чатов). Это позволяет другим пользователям видеть, что человек сейчас активен и может ответить на сообщение.

- При отключении: статус устанавливается в OFFLINE, обновляется поле lastSeen. Когда пользователь закрывает приложение или теряет соединение, система обновляет статус на OFFLINE и записывает текущее время в поле lastSeen (время последнего посещения). Эта информация важна для других пользователей - они могут видеть, когда человек был в сети в последний раз. Например, если пользователь видит "был(а) в сети 5 минут назад", он понимает, что человек недавно был активен и может скоро вернуться.

Автоматическое управление статусом через WebSocket более надежно, чем периодические запросы от клиента, так как сервер точно знает, когда соединение установлено или разорвано. Это также снижает нагрузку на сервер, так как не требуется постоянное опрашивание статуса пользователей.

## 4. МОДУЛЬ ДРУЗЕЙ (FRIENDS MODULE)

### 4.1. Описание функционала

Модуль друзей реализует систему управления контактами пользователей через механизм запросов в друзья. Эта система позволяет пользователям находить друг друга, отправлять запросы на добавление в друзья, принимать или отклонять такие запросы.

В корпоративном мессенджере система друзей играет важную роль - она позволяет пользователям формировать свой список контактов, с которыми они чаще всего общаются. Это упрощает поиск нужных людей и позволяет быстро начать разговор с часто используемыми контактами. Кроме того, система друзей может использоваться для настройки приватности - например, пользователь может разрешить отправку сообщений только от друзей.

Процесс добавления в друзья работает по принципу запросов: один пользователь отправляет запрос другому, и получатель может принять или отклонить этот запрос. Это дает пользователям контроль над тем, кто может видеть их статус и отправлять им сообщения (если включены соответствующие настройки приватности). Система предотвращает автоматическое добавление в друзья, что важно для приватности и предотвращения спама.

Поиск пользователей позволяет находить коллег по имени, фамилии или username, что особенно полезно в больших организациях, где может быть много сотрудников. Поиск работает регистронезависимо и ищет совпадения в нескольких полях одновременно, что делает его удобным и гибким.

### 4.2. Модель данных запроса в друзья

Сущность FriendRequest содержит:
- **id** (UUID) - уникальный идентификатор запроса
- **sender** (User) - отправитель запроса
- **receiver** (User) - получатель запроса
- **status** (enum: PENDING, ACCEPTED, REJECTED) - статус запроса
- **createdAt** (timestamp) - время создания
- **updatedAt** (timestamp) - время обновления

### 4.3. Реализованные endpoints

**GET /api/friends/search?q=query** - Поиск пользователей
- Выполняет поиск по username, firstName, lastName
- Использует ILIKE для регистронезависимого поиска
- Возвращает список пользователей, соответствующих запросу

**GET /api/friends** - Получение списка друзей
- Возвращает список всех пользователей, с которыми установлена дружба
- Фильтрует запросы со статусом ACCEPTED
- Определяет друга как противоположную сторону принятого запроса

**GET /api/friends/requests** - Получение входящих запросов
- Возвращает список запросов в друзья со статусом PENDING
- Показывает только запросы, где текущий пользователь является получателем
- Включает информацию об отправителе запроса

**POST /api/friends/requests** - Отправка запроса в друзья
- Создает новый запрос в друзья
- Проверяет, что пользователь не отправляет запрос самому себе
- Проверяет отсутствие существующих запросов между пользователями
- Возвращает созданный запрос

**PATCH /api/friends/requests/:requestId/accept** - Принятие запроса
- Изменяет статус запроса на ACCEPTED
- Проверяет, что текущий пользователь является получателем запроса

**PATCH /api/friends/requests/:requestId/reject** - Отклонение запроса
- Изменяет статус запроса на REJECTED
- Проверяет права пользователя на отклонение запроса

### 4.4. Бизнес-логика

Система включает несколько важных правил бизнес-логики, которые обеспечивают корректную работу функции друзей:

**Предотвращение дублирования запросов**: При попытке создать запрос между двумя пользователями система проверяет наличие существующих запросов в любом направлении. Это означает, что если пользователь A уже отправил запрос пользователю B, то пользователь B не сможет отправить запрос пользователю A до тех пор, пока первый запрос не будет обработан (принят или отклонен). Это предотвращает создание множественных запросов между одними и теми же пользователями и обеспечивает целостность данных. Если пользователь попытается отправить запрос, когда уже существует запрос в любом направлении, система вернет ошибку "Friend request already exists".

**Предотвращение самодобавления**: Система проверяет, что пользователь не пытается отправить запрос самому себе. Это может показаться очевидным, но такая проверка важна для предотвращения ошибок в коде клиента и потенциальных проблем с данными. Если пользователь попытается отправить запрос самому себе, система вернет ошибку "Cannot send friend request to yourself".

**Определение друзей**: Друзья определяются как пользователи, между которыми существует запрос в друзья со статусом ACCEPTED. При получении списка друзей система находит все принятые запросы, где текущий пользователь является либо отправителем, либо получателем, и возвращает противоположную сторону запроса как друга. Это означает, что если пользователь A отправил запрос пользователю B, и B принял его, то в списке друзей A будет пользователь B, а в списке друзей B будет пользователь A.

**Обработка отклоненных запросов**: Когда пользователь отклоняет запрос в друзья, статус запроса меняется на REJECTED. Отклоненные запросы не считаются дружбой, но они остаются в базе данных. Это позволяет системе помнить, что между пользователями уже был запрос, и предотвратить повторную отправку запроса сразу после отклонения. В будущем можно добавить логику, которая позволит повторно отправлять запросы после определенного времени или после явного действия пользователя.

## 5. МОДУЛЬ ЧАТОВ (CHATS MODULE)

### 5.1. Описание функционала

Модуль чатов является основой системы обмена сообщениями. Он обеспечивает создание и управление чатами двух типов: приватные чаты (между двумя пользователями) и групповые чаты (с множеством участников). 

Приватные чаты предназначены для личного общения между двумя пользователями. Они создаются автоматически при первом сообщении или могут быть созданы явно. Приватные чаты обычно не имеют названия (или используют имена участников как название) и предназначены для неформального общения.

Групповые чаты позволяют нескольким пользователям общаться одновременно. Они имеют название, которое задается при создании, и могут включать множество участников. Групповые чаты идеально подходят для обсуждения проектов, координации работы команды или общения в рамках отдела. Создатель группового чата может добавлять и удалять участников, управляя составом группы.

Особенностью системы является поддержка уровней секретности для чатов. Это критично для корпоративных мессенджеров, где может обрабатываться конфиденциальная информация. Уровни секретности определяют, какие действия можно выполнять с сообщениями в чате - например, сообщения из чатов с грифом "государственная тайна" или "коммерческая тайна" не могут быть пересланы в другие чаты. Это обеспечивает соответствие требованиям информационной безопасности и предотвращает утечку конфиденциальной информации.

### 5.2. Модель данных чата

Сущность Chat содержит:
- **id** (UUID) - уникальный идентификатор чата
- **name** (string) - название чата
- **type** (enum: PRIVATE, GROUP) - тип чата
- **avatar** (string, nullable) - путь к аватару чата
- **securityLevel** (enum: NONE, COMMERCIAL, STATE) - уровень секретности
- **createdBy** (User) - создатель чата
- **members** (User[]) - участники чата (Many-to-Many)
- **messages** (Message[]) - сообщения в чате (One-to-Many)
- **createdAt** (timestamp) - время создания
- **updatedAt** (timestamp) - время обновления

### 5.3. Уровни секретности

Система поддерживает три уровня секретности, которые определяют степень защиты информации в чате:

- **NONE** - обычный чат без ограничений. Это стандартный уровень для обычных рабочих чатов, где нет конфиденциальной информации. В таких чатах пользователи могут свободно пересылать сообщения, редактировать и удалять их без дополнительных ограничений. Этот уровень используется по умолчанию при создании нового чата.

- **COMMERCIAL** - коммерческая тайна. Этот уровень предназначен для чатов, где обсуждается коммерчески конфиденциальная информация: финансовые данные, бизнес-планы, стратегии развития, информация о клиентах и партнерах. В таких чатах действуют ограничения на пересылку сообщений - пользователи не могут переслать сообщение из этого чата в другой чат. Это предотвращает случайную или намеренную утечку коммерческой информации. Редактирование и удаление сообщений по-прежнему разрешены, так как это не приводит к утечке информации.

- **STATE** - государственная тайна. Это самый строгий уровень секретности для чатов с информацией, составляющей государственную тайну. В таких чатах действуют те же ограничения, что и для коммерческой тайны, но с более строгим контролем. Этот уровень используется в государственных организациях или компаниях, работающих с государственными заказами, где требуется соблюдение особых требований информационной безопасности.

Уровень секретности устанавливается при создании чата и не может быть изменен после создания (это можно реализовать в будущем с дополнительными проверками прав доступа). Это гарантирует, что уровень защиты информации не может быть случайно понижен, что могло бы привести к утечке конфиденциальных данных.

### 5.4. Реализованные endpoints

**POST /api/chats** - Создание чата
- Принимает: name, type, memberIds[], securityLevel (опционально)
- Автоматически добавляет создателя в список участников
- Создает связь Many-to-Many между чатом и участниками
- Возвращает созданный чат с полной информацией об участниках

**GET /api/chats** - Получение списка чатов пользователя
- Возвращает все чаты, в которых участвует текущий пользователь
- Включает информацию об участниках и создателе
- Используется для отображения списка чатов в интерфейсе

**GET /api/chats/:id** - Получение чата по ID
- Возвращает полную информацию о чате
- Проверяет, что текущий пользователь является участником чата
- Включает всех участников и создателя

**POST /api/chats/:chatId/members/:memberId** - Добавление участника
- Добавляет нового участника в групповой чат
- Проверяет существование чата и пользователя
- Предотвращает дублирование участников

**DELETE /api/chats/:chatId/members/:memberId** - Удаление участника
- Удаляет участника из чата
- Проверяет права доступа
- Обновляет список участников

**DELETE /api/chats/:id** - Удаление чата
- Удаляет чат и все связанные сообщения (каскадное удаление)
- Проверяет, что пользователь является участником чата

### 5.5. Безопасность и авторизация

Безопасность данных в чатах критически важна, особенно когда речь идет о конфиденциальной информации. Система реализует несколько уровней защиты:

**Проверка членства в чате**: Все операции с чатами проверяют членство пользователя в чате. Пользователь может просматривать и управлять только теми чатами, в которых он является участником. Это означает, что если пользователь попытается получить доступ к чату, в котором он не участвует, система вернет ошибку 404 (Not Found), не раскрывая информацию о существовании чата. Это обеспечивает конфиденциальность данных и предотвращает несанкционированный доступ.

**Проверка прав при удалении**: При удалении чата система проверяет не только членство пользователя, но и его роль. В текущей реализации любой участник чата может удалить чат, но в будущем можно добавить проверку, что только создатель чата или администратор может его удалить. Это предотвратит случайное удаление важных чатов другими участниками.

**Проверка прав при добавлении участников**: При добавлении нового участника в групповой чат система проверяет, что чат существует и что пользователь, которого добавляют, существует в системе. Однако в текущей реализации не проверяется, имеет ли текущий пользователь право добавлять участников (например, только создатель чата может добавлять участников). Это можно улучшить в будущем, добавив роли участников (создатель, администратор, участник) и соответствующие проверки прав.

**Защита приватных чатов**: Приватные чаты имеют особую защиту - они создаются между конкретными пользователями, и доступ к ним имеют только эти пользователи. Система предотвращает добавление третьих лиц в приватный чат (хотя технически это возможно через изменение типа чата, что требует дополнительной проверки).

Все эти меры безопасности работают вместе, чтобы обеспечить конфиденциальность общения и предотвратить утечку информации из чатов, особенно тех, которые содержат конфиденциальные данные.

## 6. МОДУЛЬ СООБЩЕНИЙ (MESSAGES MODULE)

### 6.1. Описание функционала

Модуль сообщений является центральным компонентом системы, обеспечивающим отправку, получение, редактирование и удаление сообщений. Без этого модуля мессенджер не может выполнять свою основную функцию - обмен сообщениями между пользователями.

Система поддерживает различные типы сообщений: текстовые сообщения, аудиосообщения, изображения и файлы. Это позволяет пользователям не только обмениваться текстом, но и делиться медиа-контентом, что делает общение более богатым и информативным. Например, пользователь может отправить скриншот проблемы, аудиосообщение с объяснением или документ для совместной работы.

Важной особенностью системы является отслеживание статусов доставки сообщений. Пользователи могут видеть, доставлено ли их сообщение получателю и прочитано ли оно. Это особенно важно в деловой коммуникации, где нужно убедиться, что важная информация была получена и прочитана адресатом.

Система также поддерживает пересылку сообщений между чатами, что позволяет быстро делиться информацией с другими людьми или группами. Однако эта функция ограничена для чатов с уровнями секретности, чтобы предотвратить утечку конфиденциальной информации.

Редактирование сообщений позволяет пользователям исправлять опечатки или обновлять информацию в уже отправленных сообщениях. Это делает общение более удобным, так как не нужно отправлять новое сообщение с исправлениями. Система отслеживает время редактирования и показывает пометку "(ред.)" для отредактированных сообщений, что обеспечивает прозрачность изменений.

### 6.2. Модель данных сообщения

Сущность Message содержит:
- **id** (UUID) - уникальный идентификатор сообщения
- **type** (enum: TEXT, AUDIO, IMAGE, FILE) - тип сообщения
- **content** (text, nullable) - текстовое содержимое
- **fileUrl** (string, nullable) - URL файла (для аудио, изображений, файлов)
- **forwardedFromId** (UUID, nullable) - ID пересланного сообщения
- **forwardedFrom** (Message, nullable) - связь с оригинальным сообщением
- **status** (enum: SENT, DELIVERED, READ) - статус доставки
- **sender** (User) - отправитель сообщения
- **chat** (Chat) - чат, в котором отправлено сообщение
- **createdAt** (timestamp) - время создания
- **updatedAt** (timestamp) - время обновления
- **editedAt** (timestamp, nullable) - время последнего редактирования

### 6.3. Статусы сообщений

Система использует трехуровневую систему статусов доставки сообщений, которая позволяет отправителю видеть, на каком этапе находится его сообщение:

- **SENT** - сообщение отправлено, но еще не доставлено получателю. Этот статус устанавливается сразу после создания сообщения в базе данных. На этом этапе сообщение сохранено в системе, но получатель еще не получил уведомление о нем (например, если он не подключен к WebSocket или не открыл чат). В интерфейсе это обычно отображается одной галочкой или иконкой часов.

- **DELIVERED** - сообщение доставлено получателю. Этот статус означает, что сообщение было доставлено на устройство получателя и он получил уведомление о нем. Однако это не означает, что получатель прочитал сообщение - он мог просто получить уведомление, но не открыть чат. В интерфейсе это обычно отображается двумя галочками.

- **READ** - сообщение прочитано получателем. Этот статус устанавливается, когда получатель открывает чат и загружает сообщения. Это означает, что сообщение было отображено на экране получателя, и он имел возможность его прочитать. В интерфейсе это обычно отображается двумя синими галочками или галочками другого цвета.

Статусы обновляются автоматически для обеспечения актуальности информации:

- При отправке нового сообщения все предыдущие сообщения в чате помечаются как DELIVERED. Это происходит потому, что отправка нового сообщения обычно означает, что отправитель активен и подключен к системе, поэтому логично предположить, что предыдущие сообщения были доставлены. Это оптимизирует количество запросов к базе данных и обеспечивает более быстрое обновление статусов.

- При открытии чата все сообщения помечаются как READ. Когда пользователь открывает чат и загружает сообщения, система автоматически помечает все сообщения в этом чате как прочитанные. Это происходит в момент загрузки сообщений, что означает, что пользователь видел эти сообщения на экране. Это более эффективно, чем отслеживание прокрутки и определения, какие именно сообщения были прочитаны.

Такая система статусов обеспечивает прозрачность коммуникации и позволяет пользователям понимать, получил ли адресат их сообщение и ознакомился ли он с ним. Это особенно важно в деловой коммуникации, где нужно убедиться, что важная информация была доставлена и прочитана.

### 6.4. Реализованные endpoints

**POST /api/messages** - Отправка сообщения
- Принимает: type, content (опционально), fileUrl (опционально), chatId, forwardedFromId (опционально)
- Создает новое сообщение в базе данных
- Автоматически помечает предыдущие сообщения как доставленные
- Отправляет WebSocket уведомление всем участникам чата
- Возвращает созданное сообщение с полной информацией

**GET /api/messages/chat/:chatId** - Получение сообщений чата
- Поддерживает пагинацию через параметры limit и offset
- По умолчанию: limit=50, offset=0
- Максимальный limit: 100 сообщений за запрос
- Автоматически помечает сообщения как прочитанные при загрузке
- Загружает информацию об отправителе и пересланных сообщениях
- Сортирует сообщения по времени создания (ASC)

**PATCH /api/messages/:id** - Редактирование сообщения
- Позволяет редактировать только текстовое содержимое (content)
- Проверяет, что пользователь является отправителем сообщения
- Обновляет поле editedAt при редактировании
- Отправляет WebSocket уведомление об обновлении сообщения

**DELETE /api/messages/:id** - Удаление сообщения
- Удаляет сообщение из базы данных
- Проверяет, что пользователь является отправителем сообщения
- Отправляет WebSocket уведомление об удалении сообщения

**DELETE /api/messages/chat/:chatId** - Удаление всех сообщений чата
- Удаляет все сообщения в указанном чате
- Проверяет членство пользователя в чате
- Отправляет WebSocket уведомление об удалении сообщений

**PATCH /api/messages/:id/delivered** - Пометить как доставленное
- Обновляет статус сообщения на DELIVERED
- Используется для ручного обновления статуса

**PATCH /api/messages/:id/read** - Пометить как прочитанное
- Обновляет статус сообщения на READ
- Используется для ручного обновления статуса

### 6.5. Пересылка сообщений

Пересылка сообщений - это удобная функция, которая позволяет быстро делиться информацией между чатами. Система поддерживает пересылку сообщений между чатами с сохранением информации об оригинальном отправителе:

- При создании сообщения можно указать `forwardedFromId` - ID оригинального сообщения. Когда пользователь хочет переслать сообщение, он выбирает сообщение в одном чате и указывает чат, в который нужно его переслать. Система создает новое сообщение в целевом чате, но сохраняет связь с оригинальным сообщением через поле `forwardedFromId`. Это позволяет системе отслеживать источник пересланного сообщения.

- Система загружает оригинальное сообщение и его отправителя. При создании пересланного сообщения система загружает оригинальное сообщение из базы данных вместе с информацией об его отправителе. Это позволяет сохранить контекст пересланного сообщения - кто его изначально отправил и в каком чате. Эта информация важна для понимания контекста пересланного сообщения получателями.

- При отображении пересланного сообщения показывается информация об оригинальном отправителе. В интерфейсе пересланное сообщение обычно отображается с пометкой "Переслано от [Имя отправителя]" или подобной. Это позволяет получателям понимать, от кого изначально пришло сообщение, даже если оно было переслано через несколько чатов. Это обеспечивает прозрачность и позволяет отслеживать источник информации.

- Пересылка ограничена для чатов с уровнем секретности COMMERCIAL и STATE. Это критически важное ограничение безопасности. Если чат имеет уровень секретности "коммерческая тайна" или "государственная тайна", сообщения из этого чата не могут быть пересланы в другие чаты. Это предотвращает утечку конфиденциальной информации через функцию пересылки. Система проверяет уровень секретности чата, из которого пытаются переслать сообщение, и блокирует пересылку, если уровень секретности не позволяет этого. При попытке переслать сообщение из защищенного чата пользователь получает ошибку с объяснением ограничения.

Эта функция делает систему более удобной для обмена информацией, но при этом обеспечивает защиту конфиденциальных данных через ограничения на пересылку.

### 6.6. Редактирование сообщений

Редактирование сообщений - это удобная функция, которая позволяет исправлять ошибки и обновлять информацию в уже отправленных сообщениях без необходимости отправлять новое сообщение:

- Редактирование доступно только для собственных сообщений. Это важное ограничение безопасности и логики системы. Пользователь может редактировать только те сообщения, которые он сам отправил. Это предотвращает изменение сообщений других пользователей, что могло бы привести к искажению информации или злоупотреблениям. Система проверяет, что ID отправителя сообщения совпадает с ID текущего пользователя перед разрешением редактирования. Если пользователь попытается отредактировать чужое сообщение, система вернет ошибку.

- При редактировании обновляется поле `editedAt`. Когда пользователь редактирует сообщение, система сохраняет новое содержимое и записывает текущее время в поле `editedAt`. Это позволяет отслеживать, когда сообщение было отредактировано, что важно для аудита и понимания истории изменений. Поле `editedAt` остается null для неотредактированных сообщений, что позволяет легко определить, было ли сообщение изменено.

- В интерфейсе отображается пометка "(ред.)" для отредактированных сообщений. Это обеспечивает прозрачность изменений - получатели видят, что сообщение было отредактировано. Это важно для доверия в коммуникации, так как получатели знают, что содержимое сообщения могло быть изменено после отправки. В некоторых системах также показывается время редактирования или история изменений, но в текущей реализации достаточно пометки о факте редактирования.

- Изменения синхронизируются в реальном времени через WebSocket. Когда пользователь редактирует сообщение, система отправляет WebSocket событие `message-updated` всем участникам чата. Это означает, что другие пользователи видят изменения в сообщении практически мгновенно, без необходимости обновлять страницу или перезагружать чат. Это обеспечивает актуальность информации для всех участников чата в реальном времени.

Редактирование сообщений делает общение более удобным и гибким, позволяя исправлять ошибки и обновлять информацию без загромождения чата дополнительными сообщениями с исправлениями.

### 6.7. Интеграция с WebSocket

Все операции с сообщениями синхронизируются в реальном времени:
- **new-message** - уведомление о новом сообщении
- **message-updated** - уведомление об обновлении сообщения
- **message-deleted** - уведомление об удалении сообщения
- **messages-delivered** - уведомление о доставке сообщений
- **messages-deleted** - уведомление об удалении всех сообщений чата

## 7. МОДУЛЬ ЗАДАЧ (TASKS MODULE)

### 7.1. Описание функционала

Модуль задач интегрирует систему управления задачами непосредственно в мессенджер, что позволяет создавать и отслеживать задачи прямо в контексте общения. Это особенно полезно в корпоративных мессенджерах, где обсуждение проектов часто приводит к необходимости создания задач и назначения их исполнителям.

Задачи могут создаваться из сообщений - когда в чате обсуждается какая-то задача, пользователь может создать формальную задачу прямо из этого сообщения. Это сохраняет контекст обсуждения и позволяет легко отслеживать, откуда появилась задача. Например, если в чате обсуждается необходимость исправить баг, можно создать задачу из этого сообщения, и задача будет связана с исходным обсуждением.

Задачи имеют уникальные хештеги в рамках чата (например, #кв304, #bugfix-123). Хештеги позволяют легко ссылаться на задачи в сообщениях и завершать их прямо из чата, написав сообщение с хештегом. Это делает работу с задачами более удобной и интегрированной в процесс общения. Хештеги уникальны в рамках одного чата, что позволяет использовать короткие и понятные идентификаторы без конфликтов между разными чатами.

Задачи могут быть назначены нескольким исполнителям одновременно. Это позволяет создавать задачи для команды, где несколько человек должны работать над одной задачей. Например, задача "Подготовить презентацию" может быть назначена дизайнеру, копирайтеру и менеджеру проекта, каждый из которых отвечает за свою часть работы.

Система отслеживает статусы задач (открыта, в работе, завершена, отменена), что позволяет видеть прогресс выполнения задач и управлять рабочим процессом. Пользователи могут видеть задачи, назначенные им, и задачи, которые они создали, что обеспечивает полный контроль над рабочим процессом.

### 7.2. Модель данных задачи

Сущность Task содержит:
- **id** (UUID) - уникальный идентификатор задачи
- **title** (string) - название задачи
- **description** (text, nullable) - описание задачи
- **hashtag** (string) - уникальный хештег задачи (например, #кв304)
- **status** (enum: OPEN, IN_PROGRESS, COMPLETED, CANCELLED) - статус задачи
- **chat** (Chat, nullable) - чат, к которому привязана задача
- **createdBy** (User) - создатель задачи
- **assignees** (User[]) - исполнители задачи (Many-to-Many)
- **message** (Message, nullable) - сообщение, из которого создана задача
- **dueDate** (timestamp, nullable) - срок выполнения
- **createdAt** (timestamp) - время создания
- **updatedAt** (timestamp) - время обновления

### 7.3. Реализованные endpoints

**POST /api/tasks** - Создание задачи
- Принимает: title, description (опционально), hashtag (опционально), chatId (опционально), assigneeIds[], messageId (опционально), dueDate (опционально)
- Генерирует уникальный хештег, если не указан
- Проверяет уникальность хештега в рамках чата
- Требует указания хотя бы одного исполнителя
- Может быть привязана к сообщению и чату

**GET /api/tasks/chat/:chatId** - Получение задач чата
- Возвращает все задачи, привязанные к указанному чату
- Включает информацию о создателе, исполнителях и связанном сообщении
- Сортирует по времени создания (DESC)

**GET /api/tasks/my/assigned** - Получение назначенных задач
- Возвращает все задачи, где текущий пользователь является исполнителем
- Используется для отображения задач, назначенных пользователю

**GET /api/tasks/my/delegated** - Получение созданных задач
- Возвращает все задачи, созданные текущим пользователем
- Используется для отображения задач, которые пользователь делегировал другим

**GET /api/tasks/:id** - Получение задачи по ID
- Возвращает полную информацию о задаче
- Включает все связанные данные

**PATCH /api/tasks/:id/status** - Обновление статуса задачи
- Позволяет изменить статус задачи
- Проверяет права: исполнитель, создатель или участник чата
- Обновляет статус в базе данных

**POST /api/tasks/complete** - Завершение задачи по хештегу
- Принимает chatId и hashtag
- Находит задачу по хештегу в указанном чате
- Устанавливает статус COMPLETED
- Удобно для завершения задач прямо из чата

**DELETE /api/tasks/:id** - Удаление задачи
- Удаляет задачу из базы данных
- Только создатель задачи может её удалить
- Проверяет права доступа

### 7.4. Бизнес-логика задач

Система включает несколько важных правил бизнес-логики, которые обеспечивают корректную работу системы задач:

**Уникальность хештегов**: Система обеспечивает уникальность хештегов в рамках одного чата. Это означает, что в одном чате не может быть двух задач с одинаковым хештегом. Это позволяет использовать хештеги как идентификаторы задач в контексте чата. Например, если в чате есть задача с хештегом #кв304, пользователи могут ссылаться на эту задачу в сообщениях, используя хештег, и система точно знает, о какой задаче идет речь. При создании задачи без указания хештега система автоматически генерирует уникальный хештег на основе временной метки (например, #task-1704123456789). Это гарантирует уникальность даже при одновременном создании нескольких задач.

**Обязательность исполнителей**: При создании задачи система требует указания хотя бы одного исполнителя. Это гарантирует, что каждая задача имеет ответственного за ее выполнение. Задача без исполнителя не имеет смысла, так как никто не будет знать, что нужно сделать. Если пользователь попытается создать задачу без исполнителей, система вернет ошибку "Необходимо указать хотя бы одного исполнителя".

**Проверка уникальности хештега**: При создании задачи система проверяет, не существует ли уже задача с таким же хештегом в том же чате. Если такая задача существует и имеет статус OPEN (открыта), система не позволит создать дублирующую задачу. Это предотвращает создание множественных задач с одним хештегом, что могло бы привести к путанице. Однако если старая задача уже завершена или отменена, можно создать новую задачу с тем же хештегом, так как старая задача больше не активна.

**Права доступа к задачам**: Система проверяет права доступа при работе с задачами. Пользователь может изменять статус задачи, если он является исполнителем задачи, создателем задачи или участником чата, к которому привязана задача. Это обеспечивает гибкость - любой участник обсуждения может обновить статус задачи, но при этом предотвращает несанкционированные изменения. Удаление задачи доступно только создателю задачи, что предотвращает случайное удаление важных задач другими пользователями.

**Задачи без привязки к чату**: Задачи могут быть не привязаны к чату, что позволяет создавать личные задачи пользователя. Это полезно для задач, которые не связаны с конкретным чатом или проектом. Однако для задач, привязанных к чату, система проверяет членство пользователя в чате при выполнении операций. Это означает, что пользователь должен быть участником чата, чтобы видеть и управлять задачами этого чата, что обеспечивает конфиденциальность информации о задачах.

**Завершение задач по хештегу**: Система поддерживает завершение задач прямо из чата через хештег. Пользователь может написать сообщение с хештегом задачи, и система автоматически найдет задачу и изменит ее статус на COMPLETED. Это делает работу с задачами более удобной и интегрированной в процесс общения. Например, исполнитель может написать "Исправил баг #bugfix-123", и задача автоматически будет помечена как завершенная.

## 8. МОДУЛЬ НАСТРОЕК ПРИВАТНОСТИ (SETTINGS MODULE)

### 8.1. Описание функционала

Модуль настроек приватности предоставляет пользователям контроль над тем, какую информацию о себе они хотят раскрывать и кто может с ними взаимодействовать. В современном мире цифровой приватности это критически важно - пользователи должны иметь возможность контролировать свою цифровую идентичность и защищать свою приватность.

Модуль позволяет пользователям определять, кто может отправлять им сообщения. Это важная функция для защиты от спама и нежелательных сообщений. Пользователь может разрешить отправку сообщений всем пользователям системы или только своим друзьям. Это особенно полезно в публичных мессенджерах, где любой пользователь может найти и написать другому пользователю. Ограничение сообщений только друзьями помогает защитить от спама и нежелательного общения.

Управление запросами в друзья позволяет пользователям контролировать, могут ли другие пользователи отправлять им запросы на добавление в друзья. Если пользователь отключит эту функцию, другие пользователи не смогут отправить ему запрос в друзья, что может быть полезно для пользователей, которые не хотят расширять свой список контактов.

Настройки видимости статуса позволяют пользователям контролировать, видят ли другие пользователи, когда они онлайн или офлайн. Некоторые пользователи предпочитают скрывать свой статус для большей приватности, в то время как другие хотят, чтобы их друзья знали, когда они доступны для общения.

Квитанции о прочтении показывают отправителям, прочитали ли их сообщения получатели. Некоторые пользователи предпочитают отключить эту функцию для большей приватности, так как это позволяет им читать сообщения, не уведомляя об этом отправителя. Однако это может быть неудобно для отправителей, которые хотят знать, получил ли адресат их сообщение.

### 8.2. Модель данных настроек

Сущность PrivacySettings содержит:
- **id** (UUID) - уникальный идентификатор настроек
- **user** (User) - пользователь (One-to-One)
- **whoCanMessageMe** (enum: EVERYONE, FRIENDS) - кто может отправлять сообщения
- **allowFriendRequests** (boolean) - разрешение на запросы в друзья
- **showOnlineStatus** (boolean) - показывать статус онлайн
- **showReadReceipts** (boolean) - показывать квитанции о прочтении
- **createdAt** (timestamp) - время создания
- **updatedAt** (timestamp) - время обновления

### 8.3. Реализованные endpoints

**GET /api/settings** - Получение настроек приватности
- Возвращает настройки текущего пользователя
- Автоматически создает настройки по умолчанию, если их нет
- Используется для отображения текущих настроек в интерфейсе

**PATCH /api/settings** - Обновление настроек приватности
- Позволяет обновить любые параметры приватности
- Валидирует входящие данные через DTO
- Автоматически создает настройки, если их нет
- Сохраняет изменения в базе данных

### 8.4. Настройки по умолчанию

При создании нового пользователя система автоматически создает настройки приватности со следующими значениями, которые обеспечивают открытость и удобство использования системы:

- **whoCanMessageMe**: EVERYONE (все могут отправлять сообщения). Это значение по умолчанию выбрано для обеспечения открытости системы и удобства коммуникации. В корпоративном мессенджере это особенно важно, так как сотрудники должны иметь возможность связаться друг с другом для работы, даже если они не являются друзьями. Однако пользователь может изменить это значение на FRIENDS, если хочет ограничить круг людей, которые могут ему писать.

- **allowFriendRequests**: true (разрешены запросы в друзья). По умолчанию система разрешает другим пользователям отправлять запросы в друзья. Это способствует формированию сети контактов и облегчает поиск коллег. Пользователь может отключить эту функцию, если не хочет получать запросы в друзья, но это может ограничить его возможности по расширению сети контактов.

- **showOnlineStatus**: true (показывать статус онлайн). По умолчанию статус пользователя виден другим пользователям. Это помогает коллегам понимать, доступен ли человек для общения в данный момент. Однако некоторые пользователи предпочитают скрывать свой статус для большей приватности, и они могут изменить эту настройку.

- **showReadReceipts**: true (показывать квитанции о прочтении). По умолчанию система показывает отправителям, когда их сообщения были прочитаны. Это обеспечивает прозрачность коммуникации и помогает пользователям понимать, получил ли адресат их сообщение. Однако некоторые пользователи предпочитают отключить эту функцию, чтобы иметь возможность читать сообщения без уведомления отправителя.

Эти настройки обеспечивают открытость системы по умолчанию, что способствует коммуникации и сотрудничеству. Однако пользователь может изменить их в любой момент в соответствии со своими предпочтениями приватности. Система автоматически создает настройки при первом обращении к модулю настроек, если они еще не были созданы, что гарантирует наличие настроек для всех пользователей.

## 9. МОДУЛЬ ДВУХФАКТОРНОЙ АУТЕНТИФИКАЦИИ (TWO-FACTOR MODULE)

### 9.1. Описание функционала

Модуль двухфакторной аутентификации реализует дополнительный уровень безопасности при входе в систему, который значительно повышает защиту аккаунта от несанкционированного доступа. Даже если злоумышленник узнает пароль пользователя, ему понадобится доступ к устройству пользователя (обычно смартфону) для получения одноразового кода подтверждения.

Используется стандарт TOTP (Time-based One-Time Password) для генерации одноразовых кодов. TOTP - это открытый стандарт, который используется многими популярными сервисами (Google, Microsoft, GitHub и др.). Это означает, что пользователи могут использовать любое совместимое приложение-аутентификатор, такое как Google Authenticator, Microsoft Authenticator, Authy или другие. Коды генерируются на основе секретного ключа и текущего времени, что обеспечивает их уникальность и временную валидность.

Процесс работы с двухфакторной аутентификацией включает несколько этапов: генерацию секретного ключа и QR-кода, сканирование QR-кода приложением-аутентификатором, подтверждение кода для активации 2FA, и затем использование кодов при каждом входе в систему. Это может показаться сложным, но современные приложения-аутентификаторы делают этот процесс очень простым и быстрым.

Двухфакторная аутентификация особенно важна для корпоративных мессенджеров, где может обрабатываться конфиденциальная информация. Даже если пароль будет скомпрометирован (например, через фишинг или утечку данных), злоумышленник не сможет войти в систему без доступа к устройству пользователя с приложением-аутентификатором.

### 9.2. Технические детали

Система использует библиотеку `speakeasy` для генерации секретов и проверки кодов:
- Генерирует секретный ключ длиной 32 символа
- Создает QR-код для добавления в приложение-аутентификатор (Google Authenticator, Authy и т.д.)
- Проверяет коды с допустимым отклонением ±2 периода (60 секунд)

### 9.3. Реализованные endpoints

**GET /api/two-factor/status** - Получение статуса 2FA
- Возвращает информацию о том, включена ли 2FA и есть ли сгенерированный секрет
- Используется для отображения статуса в интерфейсе настроек

**POST /api/two-factor/generate-secret** - Генерация секрета и QR-кода
- Генерирует новый секретный ключ для пользователя
- Создает QR-код в формате base64 для сканирования
- Сохраняет секрет в базе данных, но не включает 2FA до подтверждения

**POST /api/two-factor/enable** - Включение 2FA
- Принимает код подтверждения от пользователя
- Проверяет код с использованием сохраненного секрета
- Включает 2FA только после успешной проверки кода
- Защищает от случайного включения без доступа к устройству

**DELETE /api/two-factor/disable** - Отключение 2FA
- Отключает двухфакторную аутентификацию
- Удаляет секретный ключ из базы данных
- Пользователь может снова войти без кода подтверждения

### 9.4. Интеграция с модулем аутентификации

Двухфакторная аутентификация тесно интегрирована с модулем аутентификации, что обеспечивает плавный процесс входа в систему:

При входе в систему модуль аутентификации проверяет, включена ли 2FA для пользователя. Это происходит после успешной проверки email и пароля. Если у пользователя не включена 2FA, процесс входа завершается стандартным образом - пользователь получает токены доступа и может начать работу.

Если 2FA включена, требуется дополнительный код подтверждения. После успешной проверки пароля система не выдает токены сразу, а требует от пользователя ввести код из приложения-аутентификатора. Это добавляет второй уровень проверки - даже если злоумышленник знает пароль, ему нужен доступ к устройству пользователя для получения кода.

Код проверяется через метод `verifyCode` модуля two-factor. Этот метод использует библиотеку speakeasy для проверки кода с использованием сохраненного секретного ключа пользователя. Система проверяет код с допустимым отклонением ±2 периода (60 секунд), что учитывает возможные расхождения времени между сервером и устройством пользователя. Это делает систему более гибкой и удобной для пользователей.

Только после успешной проверки кода пользователь получает токены доступа. Если код неверен, система возвращает ошибку "Invalid 2FA code", и пользователь должен попробовать снова. Это предотвращает несанкционированный доступ даже при знании пароля. После успешной проверки кода процесс входа завершается стандартным образом - пользователь получает токены и может начать работу.

Такая интеграция обеспечивает безопасность без значительного усложнения процесса входа для пользователей, которые включили 2FA.

## 10. МОДУЛЬ WEBSOCKET (WEBSOCKET MODULE)

### 10.1. Описание функционала

Модуль WebSocket является критически важным компонентом системы, который обеспечивает коммуникацию в реальном времени между клиентом и сервером. Без WebSocket мессенджер не может работать в режиме реального времени - пользователям пришлось бы постоянно обновлять страницу или периодически опрашивать сервер на наличие новых сообщений, что неудобно и неэффективно.

Используется библиотека Socket.IO для реализации двусторонней связи. Socket.IO - это популярная библиотека, которая предоставляет абстракцию над WebSocket протоколом и добавляет множество полезных функций: автоматическое переподключение при потере соединения, поддержку комнат (rooms) для групповой коммуникации, fallback на long-polling для старых браузеров, и удобный API для работы с событиями.

WebSocket позволяет серверу отправлять данные клиенту без запроса от клиента. Это означает, что когда один пользователь отправляет сообщение, сервер может сразу же уведомить всех других участников чата через WebSocket, и они увидят новое сообщение практически мгновенно, без необходимости обновлять страницу или делать запросы к серверу.

Модуль WebSocket также управляет статусами пользователей - когда пользователь подключается, его статус автоматически обновляется на ONLINE, а когда отключается - на OFFLINE с обновлением времени последнего посещения. Это обеспечивает актуальность информации о статусах пользователей в реальном времени.

Система использует концепцию комнат (rooms) для организации коммуникации. Каждый чат имеет свою комнату, и когда пользователь открывает чат, он присоединяется к соответствующей комнате. Это позволяет серверу отправлять сообщения только участникам конкретного чата, а не всем подключенным пользователям, что оптимизирует использование ресурсов и обеспечивает приватность коммуникации.

### 10.2. Архитектура WebSocket

Система использует namespace `/chat` для изоляции WebSocket соединений чата. Каждый клиент подключается с указанием `userId` в параметрах аутентификации.

### 10.3. События клиента → сервера

**join-chat** - Присоединение к чату
- Клиент присоединяется к комнате чата для получения уведомлений
- Параметры: `{ chatId: string }`
- Используется при открытии чата в интерфейсе

**leave-chat** - Покидание чата
- Клиент покидает комнату чата
- Параметры: `{ chatId: string }`
- Используется при закрытии чата

**send-message** - Отправка сообщения через WebSocket
- Альтернативный способ отправки сообщений
- Параметры: `{ type, content, chatId, senderId }`
- Создает сообщение через MessagesService
- Рассылает уведомление всем участникам чата

**typing-start** - Начало печати
- Уведомляет других участников чата о начале печати
- Параметры: `{ chatId: string }`
- Используется для отображения индикатора "печатает..."

**typing-stop** - Остановка печати
- Уведомляет об остановке печати
- Параметры: `{ chatId: string }`
- Скрывает индикатор печати

**mark-as-read** - Пометить сообщение как прочитанное
- Обновляет статус сообщения через WebSocket
- Параметры: `{ messageId: string, userId: string }`

### 10.4. События сервера → клиента

**new-message** - Новое сообщение
- Рассылается всем участникам чата при создании нового сообщения
- Содержит полную информацию о сообщении
- Используется для обновления интерфейса в реальном времени

**message-updated** - Сообщение обновлено
- Рассылается при редактировании сообщения
- Содержит обновленную информацию о сообщении

**message-deleted** - Сообщение удалено
- Рассылается при удалении сообщения
- Содержит: `{ messageId: string, chatId: string }`

**messages-delivered** - Сообщения доставлены
- Уведомление о доставке сообщений
- Содержит: `{ chatId: string, userId: string }`

**messages-deleted** - Все сообщения чата удалены
- Уведомление об удалении всех сообщений чата
- Содержит: `{ chatId: string }`

**user-online** - Пользователь онлайн
- Рассылается при подключении пользователя
- Содержит: `{ userId: string }`
- Обновляет статус пользователя в интерфейсе

**user-offline** - Пользователь офлайн
- Рассылается при отключении пользователя
- Содержит: `{ userId: string, lastSeen?: string }`
- Обновляет статус и время последнего посещения

**user-typing** - Пользователь печатает
- Уведомление о начале печати пользователя
- Содержит: `{ userId: string, chatId: string }`

**user-stopped-typing** - Пользователь перестал печатать
- Уведомление об остановке печати
- Содержит: `{ userId: string, chatId: string }`

### 10.5. Управление подключениями

Управление подключениями - это важная часть модуля WebSocket, которая обеспечивает отслеживание состояния пользователей и правильную маршрутизацию сообщений:

**При подключении клиента** происходит несколько важных операций:

- Система сохраняет соответствие между socket.id и userId. Каждое WebSocket соединение имеет уникальный идентификатор (socket.id), который присваивается Socket.IO при установлении соединения. Система сохраняет соответствие между этим идентификатором и ID пользователя в памяти (в Map структуре). Это позволяет быстро находить пользователя по его socket соединению и наоборот, что необходимо для отправки персональных уведомлений и управления статусами.

- Пользователь автоматически присоединяется к своей персональной комнате (userId). Помимо комнат чатов, каждый пользователь имеет свою персональную комнату с именем, равным его userId. Это позволяет отправлять персональные уведомления конкретному пользователю, даже если он не находится в конкретном чате. Например, можно уведомить пользователя о новом запросе в друзья, даже если он не открыл соответствующий чат.

- Статус пользователя обновляется на ONLINE в базе данных. Когда пользователь подключается, система обновляет его статус в базе данных на ONLINE. Это важно для других пользователей, которые могут видеть статус этого пользователя в списке контактов или в чатах. Обновление происходит асинхронно, чтобы не блокировать процесс подключения.

- Рассылается уведомление другим пользователям о подключении. После обновления статуса система рассылает событие `user-online` всем другим подключенным пользователям. Это позволяет интерфейсу других пользователей обновить статус этого пользователя в реальном времени, показывая, что он теперь онлайн. Уведомление отправляется через broadcast, что означает отправку всем подключенным клиентам, кроме отправителя.

**При отключении клиента** происходит обратный процесс:

- Удаляется соответствие socket.id и userId. Когда соединение разрывается, система удаляет запись о соответствии из Map. Это освобождает память и предотвращает попытки отправки сообщений на несуществующее соединение.

- Статус пользователя обновляется на OFFLINE. Система обновляет статус пользователя в базе данных на OFFLINE, что отражает его текущее состояние. Это важно для других пользователей, которые должны видеть, что человек больше не доступен для общения.

- Обновляется поле lastSeen. Одновременно с обновлением статуса система записывает текущее время в поле lastSeen (время последнего посещения). Это позволяет другим пользователям видеть, когда человек был в сети в последний раз. Например, если пользователь видит "был(а) в сети 5 минут назад", он понимает, что человек недавно был активен.

- Рассылается уведомление об отключении с информацией о последнем посещении. Система рассылает событие `user-offline` всем другим подключенным пользователям с информацией о времени последнего посещения. Это позволяет интерфейсу других пользователей обновить статус и показать время последнего посещения в реальном времени.

Такое управление подключениями обеспечивает актуальность информации о статусах пользователей и правильную работу системы в реальном времени.

## 11. БЕЗОПАСНОСТЬ И АВТОРИЗАЦИЯ

Безопасность является критически важным аспектом любой системы, особенно когда речь идет о корпоративном мессенджере, где может обрабатываться конфиденциальная информация. Система реализует многоуровневую защиту, которая включает аутентификацию, авторизацию, валидацию данных и защиту на уровне ресурсов.

### 11.1. JWT Аутентификация

Система использует JWT (JSON Web Tokens) для аутентификации - современный и безопасный подход к управлению сессиями пользователей:

- Все защищенные endpoints требуют валидный access token в заголовке Authorization. Когда пользователь делает запрос к защищенному endpoint, он должен включить токен в заголовок запроса в формате `Authorization: Bearer <token>`. Сервер проверяет наличие и валидность токена перед обработкой запроса. Если токен отсутствует, истек или невалиден, сервер возвращает ошибку 401 (Unauthorized), и запрос не обрабатывается. Это гарантирует, что только аутентифицированные пользователи могут получать доступ к защищенным ресурсам.

- Токены содержат информацию о пользователе (email, id, username). Эта информация встроена в токен в виде payload, что позволяет серверу идентифицировать пользователя без обращения к базе данных при каждом запросе. Это повышает производительность системы, так как не нужно делать запрос к базе данных для каждого запроса. Однако важно понимать, что эта информация не является секретной и может быть прочитана из токена (токены кодируются в base64, но не шифруются), поэтому в токенах никогда не хранятся чувствительные данные.

- Токены подписываются секретным ключом для предотвращения подделки. Каждый токен подписывается секретным ключом сервера с использованием алгоритма HMAC SHA256. Это означает, что любое изменение токена (например, попытка изменить ID пользователя) приведет к невалидной подписи, и сервер отклонит такой токен. Это предотвращает подделку токенов злоумышленниками. Секретный ключ хранится только на сервере и никогда не передается клиентам.

- Refresh токены используются для обновления access токенов без повторного входа. Access токены имеют короткое время жизни (15 минут) для безопасности, но это означает, что пользователю пришлось бы постоянно вводить пароль. Refresh токены решают эту проблему - они имеют длительное время жизни (7 дней) и используются только для получения новых access токенов. Это обеспечивает баланс между безопасностью и удобством использования.

### 11.2. Guards и Decorators

Система использует несколько механизмов защиты:

**JwtAuthGuard** - глобальный guard для всех endpoints
- Проверяет наличие и валидность JWT токена
- Извлекает пользователя из токена и добавляет в запрос
- Использует декоратор @Public() для исключения публичных endpoints

**@CurrentUser()** - декоратор для получения текущего пользователя
- Извлекает пользователя из request после прохождения JwtAuthGuard
- Используется в контроллерах для получения информации о текущем пользователе

**@Public()** - декоратор для публичных endpoints
- Помечает endpoint как публичный, исключая проверку JWT
- Используется для регистрации и входа

### 11.3. Валидация данных

Все входящие данные валидируются через DTO (Data Transfer Objects):
- Используется библиотека class-validator для валидации
- Глобальный ValidationPipe проверяет все входящие данные
- Некорректные данные отклоняются с описанием ошибок

### 11.4. Хеширование паролей

Пароли хранятся в захешированном виде:
- Используется алгоритм bcrypt с 10 раундами хеширования
- Пароли никогда не передаются и не хранятся в открытом виде
- При обновлении пароля автоматически генерируется новый хеш

### 11.5. Авторизация на уровне ресурсов

Авторизация на уровне ресурсов - это проверка прав пользователя на выполнение конкретных операций с конкретными ресурсами. Это критически важно для предотвращения несанкционированного доступа к данным других пользователей:

- Пользователи могут редактировать/удалять только свои сообщения. При попытке редактирования или удаления сообщения система проверяет, что ID отправителя сообщения совпадает с ID текущего пользователя. Если пользователь попытается изменить или удалить чужое сообщение, система вернет ошибку. Это предотвращает изменение или удаление сообщений других пользователей, что могло бы привести к искажению информации или злоупотреблениям. Например, злоумышленник не сможет удалить важное сообщение другого пользователя или изменить его содержимое.

- Доступ к чатам ограничен только участниками чата. При попытке получить доступ к чату (просмотр, получение сообщений, добавление участников) система проверяет, является ли пользователь участником этого чата. Если пользователь не является участником, система возвращает ошибку 404 (Not Found), не раскрывая информацию о существовании чата. Это обеспечивает конфиденциальность - пользователи не могут узнать о существовании чатов, в которых они не участвуют, и не могут получить доступ к их содержимому.

- Задачи могут редактироваться только исполнителями или создателями. При изменении статуса задачи система проверяет, является ли пользователь исполнителем задачи, создателем задачи или участником чата, к которому привязана задача. Это обеспечивает гибкость - любой участник обсуждения может обновить статус задачи, но при этом предотвращает несанкционированные изменения задач пользователями, которые не имеют отношения к задаче.

- Удаление задач доступно только создателям. Удаление задачи - это критическая операция, которая необратима. Поэтому система разрешает удаление задач только их создателям. Это предотвращает случайное или злонамеренное удаление важных задач другими пользователями. Даже исполнители задачи не могут ее удалить, что защищает от случайной потери важной информации.

Такая многоуровневая система авторизации обеспечивает защиту данных на всех уровнях системы и предотвращает несанкционированный доступ к информации.

## 12. БАЗА ДАННЫХ И МИГРАЦИИ

### 12.1. Структура базы данных

Система использует PostgreSQL как основную базу данных - надежную реляционную базу данных с открытым исходным кодом, которая обеспечивает целостность данных, поддержку транзакций и сложных запросов. PostgreSQL была выбрана благодаря ее надежности, производительности и богатому набору функций, которые необходимы для сложной системы с множественными связями между сущностями.

Структура базы данных включает следующие основные таблицы, каждая из которых отвечает за хранение определенного типа данных:

- **users** - пользователи системы. Эта таблица хранит всю информацию о пользователях: их учетные данные (email, username, захешированный пароль), личную информацию (имя, фамилия, биография), профессиональную информацию (должность, отдел, текущий проект), настройки безопасности (секрет для 2FA, флаг включения 2FA) и статус (онлайн/офлайн, время последнего посещения). Эта таблица является центральной в системе, так как большинство других таблиц ссылаются на пользователей.

- **chats** - чаты (приватные и групповые). Таблица хранит информацию о чатах: название, тип (приватный или групповой), уровень секретности, создателя чата и временные метки создания и обновления. Каждый чат связан с создателем через внешний ключ и с участниками через промежуточную таблицу.

- **chat_members** - связь Many-to-Many между чатами и пользователями. Это промежуточная таблица, которая реализует связь "многие ко многим" между чатами и пользователями. Один пользователь может участвовать в множестве чатов, и один чат может иметь множество участников. Таблица хранит только связи между чатами и пользователями без дополнительной информации.

- **messages** - сообщения. Таблица хранит все сообщения в системе: содержимое сообщения, тип (текст, аудио, изображение, файл), статус доставки, отправителя, чат, в котором отправлено сообщение, информацию о пересылке и редактировании, и временные метки. Каждое сообщение связано с отправителем и чатом через внешние ключи, что обеспечивает целостность данных.

- **friend_requests** - запросы в друзья. Таблица хранит запросы на добавление в друзья: отправителя, получателя и статус запроса (ожидает, принят, отклонен). Эта таблица позволяет отслеживать все запросы в друзья и определять, кто является друзьями (пользователи с принятыми запросами между ними).

- **privacy_settings** - настройки приватности. Таблица хранит настройки приватности каждого пользователя: кто может отправлять сообщения, разрешены ли запросы в друзья, показывать ли статус онлайн и квитанции о прочтении. Каждый пользователь имеет одну запись настроек приватности (связь One-to-One).

- **tasks** - задачи. Таблица хранит информацию о задачах: название, описание, хештег, статус, чат (если привязана к чату), создателя, связанное сообщение и срок выполнения. Задачи могут быть привязаны к чатам или существовать независимо.

- **task_assignees** - связь Many-to-Many между задачами и исполнителями. Это промежуточная таблица, которая реализует связь "многие ко многим" между задачами и пользователями-исполнителями. Одна задача может иметь множество исполнителей, и один пользователь может быть исполнителем множества задач.

### 12.2. Миграции

Система включает следующие миграции:

**001_add_security_level_to_chats.sql** - Добавление уровня секретности к чатам
- Добавляет поле `security_level` с возможными значениями: none, commercial, state
- Устанавливает значение по умолчанию 'none'

**002_create_tasks_table.sql** - Создание таблицы задач
- Создает таблицу `tasks` со всеми необходимыми полями
- Создает таблицу связи `task_assignees` для Many-to-Many связи

**003_add_two_factor_to_users.sql** - Добавление полей для 2FA
- Добавляет поля `twoFactorSecret` и `twoFactorEnabled` в таблицу users

**004_add_user_profile_fields.sql** - Расширение профиля пользователя
- Добавляет поля: bio, position, department, currentProject

**005_add_last_seen_to_users.sql** - Добавление поля lastSeen
- Добавляет поле `lastSeen` для отслеживания времени последнего посещения

**006_add_forwarded_from_to_messages.sql** - Поддержка пересылки сообщений
- Добавляет поле `forwardedFromId` для связи с оригинальным сообщением

**007_add_edited_at_to_messages.sql** - Поддержка редактирования сообщений
- Добавляет поле `editedAt` для отслеживания времени редактирования

### 12.3. Связи между таблицами

Система использует следующие типы связей:
- **One-to-Many**: User → Chat (createdBy), Chat → Message, User → Message (sender)
- **Many-to-Many**: Chat ↔ User (members), Task ↔ User (assignees)
- **One-to-One**: User ↔ PrivacySettings
- **Many-to-One**: Message → Message (forwardedFrom)

Все связи настроены с каскадным удалением там, где это необходимо для обеспечения целостности данных.

## 15. ОБРАБОТКА ОШИБОК

### 15.1. Типы ошибок

Система использует стандартные HTTP коды статусов:
- **200** - Успешный запрос
- **201** - Ресурс создан
- **400** - Некорректный запрос (валидация данных)
- **401** - Неавторизован (отсутствует или невалидный токен)
- **403** - Доступ запрещен (недостаточно прав)
- **404** - Ресурс не найден
- **409** - Конфликт (дублирование данных)
- **500** - Внутренняя ошибка сервера

### 15.2. Структура ответа об ошибке

Все ошибки возвращаются в стандартизированном формате:
```json
{
  "statusCode": 400,
  "message": "Описание ошибки",
  "error": "Тип ошибки"
}
```

### 15.3. Валидация данных

Ошибки валидации автоматически форматируются и возвращаются клиенту с описанием проблемных полей. Это обеспечивает понятные сообщения об ошибках для разработчиков фронтенда.

## 16. ПРОИЗВОДИТЕЛЬНОСТЬ И ОПТИМИЗАЦИЯ

### 16.1. Кэширование

Система использует кэширование для оптимизации производительности:
- Настроен глобальный кэш с TTL 10 минут
- Максимальное количество записей в кэше: 100
- В production рекомендуется использовать Redis для распределенного кэширования

### 16.2. Пагинация

Все endpoints, возвращающие списки данных, поддерживают пагинацию:
- Параметры `limit` и `offset` для управления количеством данных
- Ограничение максимального размера выборки (например, 100 сообщений за запрос)
- Предотвращение перегрузки базы данных большими запросами

### 16.3. Оптимизация запросов к БД

Система использует TypeORM QueryBuilder для оптимизации запросов:
- Eager loading связанных сущностей только при необходимости
- Использование leftJoinAndSelect для загрузки связанных данных одним запросом
- Избежание N+1 проблем через правильное использование relations

## ЗАКЛЮЧЕНИЕ

В рамках данной работы была реализована полнофункциональная бэкенд-система для веб-приложения обмена сообщениями, которая представляет собой современное и масштабируемое решение для корпоративной коммуникации. Система включает в себя все необходимые модули для обеспечения работы современного мессенджера с поддержкой реального времени, безопасности и интеграции с системами управления задачами.

Система состоит из восьми основных модулей, каждый из которых решает конкретные задачи:

1. **Модуль аутентификации** с поддержкой JWT токенов и двухфакторной аутентификации обеспечивает безопасный вход в систему и защиту от несанкционированного доступа. Использование двух типов токенов (access и refresh) обеспечивает баланс между безопасностью и удобством использования.

2. **Модуль управления пользователями** с расширенными профилями и управлением статусами позволяет пользователям создавать подробные профили с профессиональной информацией и автоматически отслеживать статусы онлайн/офлайн в реальном времени.

3. **Модуль друзей** с системой запросов в друзья обеспечивает управление контактами и формирование сети связей между пользователями, что упрощает коммуникацию и поиск нужных людей.

4. **Модуль чатов** с поддержкой приватных и групповых чатов и уровнями секретности позволяет создавать различные типы чатов для разных целей и обеспечивает защиту конфиденциальной информации через уровни секретности.

5. **Модуль сообщений** с поддержкой различных типов сообщений, пересылки, редактирования и удаления является центральным компонентом системы, обеспечивающим богатую коммуникацию с отслеживанием статусов доставки и возможностью исправления ошибок.

6. **Модуль задач** для управления задачами в контексте чатов интегрирует управление задачами непосредственно в процесс общения, что делает работу с задачами более удобной и контекстной.

7. **Модуль настроек приватности** для управления параметрами безопасности дает пользователям контроль над своей цифровой приватностью и определяет, кто может с ними взаимодействовать.

8. **Модуль WebSocket** для коммуникации в реальном времени обеспечивает мгновенную синхронизацию данных между всеми подключенными клиентами, что критично для современного мессенджера.

Система построена на современных технологиях и следует лучшим практикам разработки:

- Модульная архитектура с четким разделением ответственности обеспечивает поддерживаемость и масштабируемость кода. Каждый модуль инкапсулирует свою логику и может разрабатываться и тестироваться независимо.

- Использование TypeScript для типобезопасности позволяет выявлять ошибки на этапе компиляции и улучшает читаемость кода, что особенно важно при работе в команде.

- RESTful API с автоматической документацией через Swagger упрощает интеграцию с фронтендом и позволяет разработчикам быстро понимать доступные endpoints и тестировать их.

- WebSocket для реального времени обеспечивает мгновенную коммуникацию без необходимости постоянного опрашивания сервера, что улучшает пользовательский опыт и снижает нагрузку на сервер.

- Контейнеризация через Docker для простого развертывания обеспечивает консистентность работы в различных окружениях и упрощает процесс развертывания и масштабирования.

- Безопасность через JWT, хеширование паролей и авторизацию на уровне ресурсов обеспечивает многоуровневую защиту данных и предотвращает несанкционированный доступ.

Система готова к использованию в production окружении и может быть масштабирована для поддержки большого количества пользователей через добавление балансировщиков нагрузки, распределенных систем кэширования (Redis), горизонтального масштабирования через несколько экземпляров приложения и оптимизацию запросов к базе данных.

Дальнейшее развитие системы может включать добавление новых функций, таких как голосовые и видеозвонки, интеграция с внешними системами (email, календари), расширенная аналитика использования, мобильные приложения и улучшенная система уведомлений. Архитектура системы позволяет легко добавлять новые модули и функции без нарушения существующей функциональности.
